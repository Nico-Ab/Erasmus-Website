# Testing Strategy

## Purpose
This document defines how automated testing should be used in the Erasmus staff mobility portal so the project remains maintainable, locally runnable, and safe to evolve.

## Testing goals
- Catch regressions early in shared logic and validation.
- Verify component behavior through user-visible outcomes rather than implementation details.
- Keep at least one reliable end-to-end smoke path working from the browser.
- Support local-first development with tests that can be run on a developer machine without cloud services.
- Avoid brittle snapshot-heavy tests that add noise without confidence.

## Test layers
### Unit tests
Use Vitest for:
- validation schemas
- formatting helpers
- navigation logic
- other pure functions with minimal dependencies

Characteristics:
- fast
- deterministic
- no network or database dependency
- stored under `tests/unit/`

### Integration tests
Use Vitest with React Testing Library for:
- form behavior
- component interaction with validation and mocked side effects
- route-adjacent components that depend on Next.js hooks or Auth.js client functions

Characteristics:
- run in `jsdom`
- use mocks only for framework edges such as router/auth hooks
- assert user-facing behavior and side effects
- stored under `tests/integration/`

### Component tests
Use React Testing Library for:
- presentational pages or reusable UI blocks
- conditional rendering based on props
- primary calls to action and visible content

Characteristics:
- focus on what users can see and trigger
- avoid snapshot-only coverage
- stored under `tests/components/`

### End-to-end smoke tests
Use Playwright for:
- basic route availability
- login reachability
- authenticated app shell rendering

Characteristics:
- run against the local app at `http://localhost:3000`
- can start the local dev server automatically through Playwright
- should stay small, stable, and high-signal
- stored under `tests/e2e/`

## Current local commands
- `npm run test`: runs Vitest unit, integration, and component suites
- `npm run test:watch`: runs Vitest in watch mode
- `npm run test:e2e`: runs Playwright smoke coverage
- `npm run test:e2e:headed`: runs Playwright with a visible browser window

## Local prerequisites for E2E
Before Playwright smoke tests, the local environment should have:
- PostgreSQL running through Docker
- Prisma migration applied
- seeded test users present
- Node.js available on the host machine

Playwright is configured to reuse an already running local server when available. If no local server is running, it will start one with `npm run dev`.

## Helpers and factories
Current shared testing utilities:
- `tests/helpers/render.tsx`: wraps React Testing Library render with `userEvent`
- `tests/factories/auth.ts`: provides stable credentials fixtures for login-related tests

Guidance:
- add helpers when they remove repetition across multiple tests
- add factories for stable domain inputs, not random data
- prefer explicit fixture values over fuzzy autogenerated test data for administrative workflows

## What is covered now
### Unit coverage
- login schema trimming and validation rules
- role-aware navigation filtering
- shared formatting helpers

### Integration coverage
- login form validation before submit
- successful credentials submit flow with router redirect
- failed credentials submit flow with user-visible error state

### Component coverage
- home page anonymous rendering
- home page authenticated call-to-action rendering

### E2E smoke coverage
- home page loads
- login page loads
- authenticated dashboard shell renders after sign-in

## What remains uncovered
Critical flows still needing future coverage:
- registration and approval workflow
- staff profile editing
- mobility case draft and submission lifecycle
- protected document upload/download authorization
- officer review actions and status transitions
- admin user and master data management
- reporting and CSV export behavior
- audit-sensitive server actions

## Test authoring rules
- Test behavior, not implementation trivia.
- Prefer role, label, and text queries over brittle selectors.
- Keep unit tests pure and fast.
- Keep integration tests narrowly scoped to one behavior cluster.
- Keep E2E smoke tests short and resilient.
- Do not rely on large snapshots to prove correctness.
- When a bug is fixed, add the smallest test that would have caught it.
